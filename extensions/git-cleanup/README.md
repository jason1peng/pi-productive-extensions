# git-cleanup

Adds `/cleanup` for the common post-merge repo housekeeping flow.

For compatibility with RPC clients that wait for Pi's agent lifecycle, `/cleanup` starts an agent turn that invokes the `git_cleanup` tool. This also makes RPC agent cancellation reach the Git process. The extra model turn is a temporary workaround for the [Paseo extension-command lifecycle issue](../../docs/paseo-extension-command-lifecycle.md).

The tool:

1. Requires `/cleanup` to be invoked from the repository's stable primary checkout. Invocation from a linked worktree fails with the path of the primary checkout to use instead.
2. Checks that `main` is not checked out in a linked worktree. If it is, cleanup stops with instructions to switch or remove that worktree; it never silently leaves the primary checkout on a planning branch.
3. Preflights `git status --porcelain` for every worktree before fetching or mutating anything. Any status failure aborts cleanup and preserves every worktree and branch rather than treating an unreadable status as clean.
4. In a live run, fetches `origin/main` directly into a unique operation-private ref, validates and pins that fetched commit OID, and uses it for every ancestry, patch-equivalence, merge, and cleanup decision. A concurrent process can move `origin/main` after the fetch without changing this invocation's provenance-safe target. The private ref is compare-and-swap deleted in `finally` on success or failure. If fetch fails before its OID is pinned, cleanup uses `git show-ref --verify` and treats only its documented missing-ref exit as verified absence; an inspection failure fails closed. Failure to inspect or remove the ref is never suppressed: the error names the possibly leaked ref and manual recovery commands, and if the main operation also failed an `AggregateError` reports both failures. In a dry-run, reads the remote branch OID with `git ls-remote` and downloads objects without writing `FETCH_HEAD` or updating any local/remote-tracking ref, so decisions use current remote state even when `origin/main` is stale.
5. If the primary checkout is temporarily on a merged or patch-equivalent `plan/<slug>` branch with no tracked changes, switches it back to `main` with Git's `--no-overwrite-ignore` protection, fast-forwards it to current remote main, and deletes the displaced planning branch. Before the fast-forward it intersects changed paths with both ordinary-untracked and ignored paths, including directory/file conflicts, so the merge cannot knowingly replace local content. Untracked-only primary state, especially `.pi-subagents/` runtime artifacts, is otherwise permitted and preserved. Automatic primary restoration/deletion is deliberately limited to the planning-branch convention used by the planning workflow. Any detached HEAD or branch outside `plan/*` is preserved with instructions to switch back manually; planning branches with tracked changes or unmerged commits are also preserved.
6. Otherwise, performs the same untracked/ignored overwrite preflight and fast-forwards `main` in the primary checkout to the pinned remote-main commit.
7. Removes clean non-main linked worktrees whose HEAD is already merged into the pinned remote-main commit or whose merge-free candidate range is patch-equivalent to commits already there (for rebased/cherry-picked MR merges). Candidate ranges containing merge commits are never accepted through patch equivalence because `git cherry` cannot prove that unique conflict-resolution content is present upstream. Cleanup deletes only the recognized `.pi-subagents/` runtime-artifact directory, performs an ignored-file scan, and then moves the registered worktree atomically to an unguessable sibling quarantine path. A second status plus ordinary-untracked/ignored inspection occurs in quarantine. If content arrived after the prior scan, cleanup moves the worktree back, preserves its branch, and reports it as skipped. Only an empty, unchanged quarantine is passed to normal (non-`--force`) `git worktree remove`; a refusal is also skipped with the quarantine and branch preserved.
8. Deletes the matching local branch after the worktree is removed using an expected-HEAD compare-and-swap. If the branch moves after eligibility was checked, cleanup fails clearly and preserves the moved branch instead of force-deleting new work.
9. Runs `git worktree prune` and leaves the primary checkout on `main`.

Git commands run asynchronously through Pi's abortable process API. Local commands time out after 15 seconds, network fetch commands after 60 seconds, and interactive credential prompts are disabled. The tool streams progress and clears its status on success, failure, or cancellation.

It skips linked worktrees with tracked modifications, newly discovered content, quarantine/move failures, and linked worktrees not merged or safely patch-equivalent to current remote main; one affected linked worktree does not abort cleanup of safe peers. A linked worktree containing only recognized `.pi-subagents/` runtime artifacts can be removed. The primary-checkout restoration is a special case: it blocks tracked changes but permits and preserves non-conflicting untracked-only state, requires a fast-forwardable local `main` and an eligible `plan/<slug>` branch, and protects ignored files during both the switch and fast-forward. It never removes the primary checkout or automatically displaces/deletes an arbitrary long-lived branch.

### Concurrency boundary

The quarantine rename is the linked-worktree publication boundary: after it succeeds, normal supported writers must resolve and open files through the worktree's published original path. Such a writer either reaches the worktree before the move (and its content moves into quarantine for the final inspection) or creates/leaves content at the now-unregistered original path, which cleanup never deletes. Processes that retain an already-open directory descriptor or working directory and continue mutating it after it has been renamed are outside the supported concurrency model; no portable Git/filesystem primitive can freeze those writers atomically with inspection and deletion. Likewise, callers must not concurrently mutate the primary checkout during its final overwrite preflight and fast-forward. Under this explicit boundary, ignored arrivals are not silently deleted; ambiguous or detected changes fail closed and preserve the worktree/branch.

The implementation requires a Git version whose `git switch` supports `--no-overwrite-ignore`. Unsupported Git fails before changing the primary checkout rather than silently falling back to overwrite-prone behavior.

### Private-ref recovery

Normally no `refs/pi-cleanup/targets/*` ref remains. If private-ref inspection or compare-and-swap removal fails, cleanup reports that guarantee failure explicitly, including the exact ref that may remain; it does not confuse an unreadable ref with verified absence or hide this behind an earlier operation error. Follow the reported instructions: inspect the ref with `git show-ref --verify <ref>`, verify that it is the operation-private cleanup target, obtain its current OID from that output, then remove exactly that value with `git update-ref -d <ref> <current-oid>`. If the value changes, inspect again rather than deleting it without an expected OID.

Options:

- `/cleanup --dry-run` or `/cleanup -n` — query current remote main and show the resulting plan without changing worktrees, branches, HEAD, `FETCH_HEAD`, or refs (missing commit objects may be downloaded).
- `/cleanup --main trunk` — use a different main branch name.
- `/cleanup --force-current` — retained for argument compatibility; the supported lifecycle requires invocation from the primary checkout, which is never removed.
